**LD5: Same-state PATCH returns 409 `INVALID_STATE_TRANSITION`, not 200 no-op.**

Inference based on the tenants `transition` matrix (`step-6_11-impl-2026-05-14.md`, repo precedent block):

```
allowed_sources = {
    "SUSPENDED": {"TRIAL", "ACTIVE"},
    "ACTIVE": {"TRIAL", "SUSPENDED"},
}[target_status]
if current.status not in allowed_sources:
    return None, TransitionResult.INVALID_STATE
```

Tenants does NOT include the target state in its `allowed_sources` set, so `ACTIVE -> ACTIVE` falls through to INVALID_STATE. Stores `TRANSITION_MATRIX` should mirror this convention: target state NOT in its own allowed-sources set; same-state returns INVALID_STATE.

**Pre-flight verify (Check P6):** confirm tenants ACTIVE -> ACTIVE smoke or test asserts 409, not 200. If precedent shows 200, override LD5 to idempotent-200.

Owner: inference from tenants precedent; pre-flight verify required.

**LD6: URL is `POST /stores/{store_id}/set-status`.** Hyphenated (codebase convention from openapi.json: multi-word verbs hyphenated per `fleet-stats`, `governance-stats`, `org-tree`). POST not PATCH (codebase convention from openapi.json: every state-transition action endpoint is POST per `/tenants/{id}/suspend`, `/tenants/{id}/activate`, `/tenant-users/{id}/suspend`, `/tenant-users/{id}/activate`, `/module-access/{tid}/{module}/enable`, `/module-access/{tid}/{module}/disable`).

Note: stores **deliberately diverges** from the per-action-endpoint pattern (tenants ships separate `/suspend` and `/activate` URLs). Stores has 4 states and 9 valid transitions; 4+ separate URLs would multiply endpoint count without clarity. Single consolidated `set-status` endpoint takes `target_status` in body. This is a one-resource deviation, not a new project-wide convention. Owner: operator decision (6.17 root design); openapi.json enumeration confirms hyphenation + POST.

**LD7: Pattern (b) audit-actor population on every write.** Reused verbatim from 6.17.3 LD9. UPDATE statement populates `updated_by_user_id` + `updated_by_user_type` on every transition. Class 1 (into-CLOSED) additionally populates `closed_by_user_id` + `closed_by_user_type`. Class 2 (out-of-CLOSED) additionally nulls the closed_by pair. Class 3 (between non-CLOSED) leaves closed_* unchanged. `_actor_type_from_auth(auth)` bridges AuthContext to ActorUserType. Owner: 6.17.3 LD9 precedent.

### repositories/stores.py (MODIFY: +transition method + TRANSITION_MATRIX constant)

**Module-level constant** (placed near `SORT_MAP`):

```
TRANSITION_MATRIX: dict[StoreStatus, set[StoreStatus]] = {
    StoreStatus.OPENING:  {StoreStatus.ACTIVE, StoreStatus.INACTIVE, StoreStatus.CLOSED},
    StoreStatus.ACTIVE:   {StoreStatus.INACTIVE, StoreStatus.CLOSED},
    StoreStatus.INACTIVE: {StoreStatus.ACTIVE, StoreStatus.CLOSED},
    StoreStatus.CLOSED:   {StoreStatus.ACTIVE, StoreStatus.INACTIVE},
}
```

Each key maps to the set of allowed target states. Same-state (e.g., ACTIVE -> ACTIVE) is NOT in any allowed set, so falls through to INVALID_STATE. Mirrors tenants pattern cited in LD5.

**`transition` method** (~40 lines):

```
async def transition(
    self,
    session: AsyncSession,
    store_id: UUID,
    *,
    target_status: StoreStatus,
    actor_user_id: UUID,
    actor_user_type: ActorUserType,
) -> tuple[StoreDetailRow | None, TransitionResult]:
    ...
```

Body shape:

1. `SELECT id, status FROM core.stores WHERE id = :store_id FOR UPDATE`. If no row, return `(None, TransitionResult.NOT_FOUND)`.
2. Check `target_status in TRANSITION_MATRIX[current.status]`. If not, return `(None, TransitionResult.INVALID_STATE)`.
3. Build the UPDATE SQL based on direction:
   - **Class 1 (into-CLOSED):** `status` + `closed_at = now()` + `closed_by_user_id = :actor_id` + `closed_by_user_type = CAST(:actor_type AS core.actor_user_type_enum)` + `updated_at = now()` + `updated_by_user_id = :actor_id` + `updated_by_user_type = CAST(:actor_type AS core.actor_user_type_enum)`.
   - **Class 2 (out-of-CLOSED):** `status` + `closed_at = NULL` + `closed_by_user_id = NULL` + `closed_by_user_type = NULL` + `updated_at = now()` + `updated_by_user_id = :actor_id` + `updated_by_user_type = CAST(:actor_type AS core.actor_user_type_enum)`.
   - **Class 3 (between non-CLOSED):** `status` + `updated_at = now()` + `updated_by_user_id = :actor_id` + `updated_by_user_type = CAST(:actor_type AS core.actor_user_type_enum)`. Closed_* columns untouched.
4. Re-fetch the row joined with `tenants` (mirror `get_by_id` shape) and return `(row, TransitionResult.OK)`.

Schema-qualify all raw SQL (`core.stores`).

Import `TransitionResult` from wherever pre-flight Check #7 identifies it (likely `from admin_backend.repositories.tenants import TransitionResult` or `_common`); if it lives elsewhere, adapt.

### routers/v1/stores.py (MODIFY: +1 handler)

```
@router.post("/{store_id}/set-status", response_model=StoreDetail)
async def set_store_status(
    store_id: UUID,
    body: StoreSetStatusRequest,
    _: None = Depends(require(
        ModuleCode.ADMIN, PermissionResource.STORES,
        PermissionAction.CONFIGURE, PermissionScope.TENANT,
        anchor_dep=get_store_anchor,
    )),

) -> StoreDetail: ...
```

Handler body:

```
row, result = await _repo.transition(
    session, store_id,
    target_status=body.target_status,
    actor_user_id=auth.user_id,
    actor_user_type=_actor_type_from_auth(auth),
)
if result is TransitionResult.NOT_FOUND:
    raise StoreNotFoundError(store_id)
if result is TransitionResult.INVALID_STATE:

        resource="store",
        store_id=store_id,
        target_status=body.target_status,
    )
return _detail_from_row(row)
```
