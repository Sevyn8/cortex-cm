# Doc Replacement Investigation Prompt

**Purpose:** Investigate impact of replacing currently-in-use CLAUDE.md, BUILD_PLAN.md, and architecture.md with new versions in `docs/archive/`. Load-bearing investigation; verdict drives adopt / adopt-with-edits / do-not-adopt decision.

**Usage:** Paste the contents of the fenced block below into Claude Code.

---

```
INVESTIGATION TASK — LOAD-BEARING. Read this entire prompt before starting.

I'm considering replacing three currently-in-use files at the repo root with new versions located in docs/archive/:

- CLAUDE.md → docs/archive/CLAUDE_v2.md
- BUILD_PLAN.md → docs/archive/BUILD_PLAN_v3.md
- architecture.md → docs/archive/architecture_v2.md

The new versions were generated by Claude Chat to (a) correct drift, (b) tighten structure, AND (c) reflect a CHANGE IN PROJECT SCOPE — the new files describe the project's current/intended reality, which may be narrower, broader, or differently-shaped than what the old files describe. They have NOT been verified against the actual codebase or deploy pipeline.

My decision after your investigation: adopt-as-is, adopt-with-edits (you supply the edits), or do-not-adopt. The investigation is load-bearing — I will act on what you report. Do not cut corners.

=== NEUTRALITY MANDATE — READ TWICE ===

Your job is to report what is TRUE, not what is convenient. The following biases are known failure modes — explicitly guard against each:

1. SUNK-COST BIAS — the new files exist and represent prior effort. This is IRRELEVANT to whether they should be adopted. Do not factor "work already done" into recommendations.

2. EFFORT-AVOIDANCE BIAS — if adopting a file would require rewriting code, adding migrations, redoing deploy steps, or other downstream work, this is NOT a reason to soften a finding or recommend "adopt-with-edits" instead of "do-not-adopt." Surface the effort honestly; do not let it bend the verdict.

3. REJECTION-AS-SAFETY BIAS — recommending "do-not-adopt" because rejection feels safer than approval is equally wrong. If the new file is genuinely better, say so even if adoption means work.

4. TIMELINE BIAS — do not consider "how long this took to produce" or "we're behind schedule" or "shipping pressure." None of that is in scope. Truth about doc correctness is the only metric.

5. AUTHORITY BIAS — do not defer to the current files because they're "in use" nor to the new files because they're "newer and tightened." Both are candidates being weighed against code reality. Code is ground truth; everything else is candidate.

6. SCOPE-CHANGE-DEFERENCE BIAS — the fact that the new files reflect a redefined scope is NOT a reason to accept the redefinition uncritically. The new scope is itself a claim to be checked against deployed reality. A scope-redefinition document is not self-justifying.

7. SYMMETRIC-BUCKET BIAS — do not pad GOOD to match BAD or vice versa to look "balanced." If a file is 90% improvement and 10% regression, say so. If it's the inverse, say so. Asymmetric findings are normal.

8. NICE-NUMBER BIAS — do not round verdicts toward "adopt-with-edits" because it sounds like a moderate compromise. Pick the verdict the evidence supports, even if that's a hard adopt-as-is or a hard do-not-adopt.

9. NON-TECHNICAL-SOFTENING — known failure mode in chat models: when findings touch on workflow, process, or human-readable convention (rather than hard technical facts), responses tend to soften. Apply the same rigor to conventions, workflow steps, and prose claims as to code claims. A wrong gcloud flag and a wrong workflow rule are equally findings.

10. EFFORT-AND-FINDINGS-MUST-BE-SEPARATE — first establish what is true. Only after findings are locked may you estimate adoption effort. Do not let effort estimation flow backward and edit findings.

If at any point you catch yourself drafting a softer finding because the harder one would imply more downstream work, STOP and write the harder finding. Then in the Self-Report section, note that you caught and corrected a bias instance.

=== HARD RULES ===
- Investigation only. Do not modify, move, rename, or delete any file under the repo's source tree. Do not stage anything. Do not run deploy or migration commands. Read-only operations only (view, grep, search, git log/blame are fine).
- THE ONLY WRITE YOU PERFORM is creating the final report file (see OUTPUT below). No other writes.
- CODE IS GROUND TRUTH. When the docs (old OR new) disagree with the code, the code wins. Both doc versions are candidates being evaluated against code reality.
- EVIDENCE REQUIRED. Every assertion in your output must have a file path + line number citation, OR be explicitly marked "unable to verify — reason: X". No assertions from memory.
- DO NOT FABRICATE TO FILL BUCKETS. If a section has no findings, write "None found" and move on.
- DO NOT SOFTEN. State findings as facts. Use "this rule contradicts X at path:line" not "this rule may possibly conflict with X."
- FLAG SILENCES. A rule, section, or convention present in the OLD version and absent from the NEW version is a finding even if nothing was "changed."

=== PHASE 0: PRE-FLIGHT ===
Confirm all six files exist at the stated paths. List each path and "EXISTS" / "MISSING". If any are missing or are at different paths, STOP and report.

Also confirm the /reports directory exists at the repo root. If it does not exist, create it (this is the one creation operation allowed before the report write). If creating /reports would conflict with anything, stop and ask before proceeding.

=== PHASE 1: BUILD AN INVENTORY OF REALITY ===
Before reading any of the six docs, build a ground-truth inventory of the codebase.

Enumerate and list (with paths):
- Every router file
- Every SQLAlchemy model and the schema it belongs to
- Every alembic migration file and what it does in one line
- Every DDL file at repo root and what schema/tables it defines
- Every repository file under repositories/ and which entity it serves
- Cloud Run service config: name, region, project, env vars referenced in deploy steps
- Every test file and what it covers (one-line summary)
- Git log on the three CURRENT files (top 10 commits each)

Include this inventory in the report.

=== PHASE 2: READ ALL SIX FILES IN FULL ===
Read all six files end-to-end. State at the end of this phase in the report: "Read all six files in full."

=== PHASE 3: SCOPE-CHANGE ANALYSIS ===
For each of the three new files, extract every scope-level claim and classify it:

A. IN-SCOPE CLAIMS WITH CODE BACKING — claim + cite the router/model/migration/DDL that backs it.

B. IN-SCOPE CLAIMS WITHOUT CODE BACKING (aspirational) — claim + note no backing code exists.

C. OUT-OF-SCOPE / REMOVED / DEPRECATED CLAIMS CONTRADICTED BY EXISTING CODE — the new doc says "we don't do X" but X exists. Cite the orphan code. RATE: would-break-if-removed / safe-to-remove / unclear.

D. SCOPE CLAIMS THAT ARE AMBIGUOUS — list as open questions.

E. SCOPE INCONSISTENCIES ACROSS THE THREE NEW FILES.

Tactic: for each item in the Phase 1 inventory, ask "is this in or out of scope per the new docs?" — if no doc mentions it, that's a silence to flag.

NEUTRALITY CHECK FOR THIS PHASE: a scope redefinition is not automatically valid because it's newer or more structured. Test it against deployed reality with the same rigor as any other claim.

=== PHASE 4: PER-FILE GOOD / BAD / UGLY ===
For each pair:

GOOD: Concrete improvements. Each item cites the section in the new file + supporting code evidence.

BAD: Regressions. Each item cites both versions.

UGLY: Active harm if adopted. Rated: would-break-deploy / would-cause-data-issue / would-mislead-future-edits / cosmetic.

Empty buckets are valid.

ITEMS TO CHECK (examples, not exhaustive):
- Schema-qualification rule in CLAUDE_v2 → verify against repositories/*.py. Find raw SQL anywhere in repositories/ and confirm each query is or isn't schema-qualified.
- Em-dash convention scope → check current OpenAPI generation path.
- BUILD_PLAN_v3 12-step workflow — walk EVERY command. Specifically verify: Dockerfile path/base image, Artifact Registry repo and region, Cloud Run service name (admin-backend), region (asia-south1), project (ithina-retail-admin), env-var flag is --update-env-vars NOT --set-env-vars, image deploys use sha256 digest not tag, Step 8.0 alembic pre-check is three separate commands (jobs update --args=current → execute --wait → revert to upgrade,head), log-parsing regex is `[a-f0-9]{12} \(head\)` with NO ^ anchor.
- architecture_v2 components — every service, schema, router named must exist; module enum unification from step 6.6 must be reflected.

=== PHASE 5: CROSS-FILE CONSISTENCY ===
Do the three new files contradict each other? List each contradiction with citations from both.

=== PHASE 6: KNOWN JANITORIAL ITEMS ===
Report whether the new versions preserve, address, or silently drop:
- docs/endpoints/openapi.json unicode-escape drift (\u2014 vs literal —)
- localhost IPv4 workaround for local DNS latency
- Whether the no-em-dash convention applies to auto-generated OpenAPI descriptions

=== PHASE 7: EFFORT ESTIMATION (DOWNSTREAM ONLY) ===

ONLY after Phases 0–6 are complete and findings are locked, estimate adoption effort. Effort is INFORMATIONAL — it does not modify findings or verdicts above. Treat this as a separate report consumed alongside, not in place of, the findings.

For each file:
- Effort to adopt-as-is: code changes required, migrations required, deploy-pipeline changes required (with file paths)
- Effort to adopt-with-edits: which edits, and effort to apply them
- Effort to do-not-adopt: any partial improvements worth cherry-picking into the current files instead

State clearly: "These effort estimates are informational. They were calculated AFTER findings were locked and did not influence any GOOD/BAD/UGLY classification or verdict."

=== PHASE 8: SELF-REPORT ===
State explicitly:
- Which directories you grep'd / examined
- Which directories you did NOT examine and why
- Which files you read in full vs. partially
- Any tools or queries that failed
- Confidence level (high/medium/low) for each major finding category, with rationale
- BIAS AUDIT: list any moments during the investigation where you caught yourself softening a finding, rounding a verdict, or weighing effort against truth — and how you corrected. If you caught no such moments, say so explicitly, and re-scan your draft once more before submitting.
```
