### NEW: Alembic migration

A new file under `migrations/versions/` (Alembic generates the hash). Filename pattern matches Step 6.20.3's: `<hash>_step_6_16_1_audit_log_schema.py`.

The migration's `upgrade()`:

1. Capture schema via `bind.execute(sa.text("SELECT current_schema()")).scalar_one()` per the CSD-03 pattern.
2. `CREATE TYPE {schema}.audit_result_type_enum AS ENUM ('SUCCESS', 'PERMISSION_DENIED', 'VALIDATION_FAILED', 'CONFLICT', 'INTEGRITY_VIOLATION', 'INTERNAL_ERROR');`
3. `CREATE TABLE {schema}.tenant_activity_audit_logs ( ... 16 columns ... );` per the design doc Table 1 spec. All columns NOT NULL except `resource_id` and `resource_label`.
4. `ALTER TABLE {schema}.tenant_activity_audit_logs ADD CONSTRAINT fk_tenant_activity_audit_logs_tenant FOREIGN KEY (tenant_id) REFERENCES {schema}.tenants(id) ON UPDATE RESTRICT ON DELETE RESTRICT;`
5. Two CHECK constraints on tenant table (named `ck_tenant_activity_audit_logs_resource_pair`): one enforces `(resource_id IS NULL AND resource_label IS NULL) OR (resource_id IS NOT NULL AND resource_label IS NOT NULL)`. Note: tenant table does NOT need the tenant_id/tenant_name NULL-pair CHECK since both are NOT NULL.
6. `ALTER TABLE {schema}.tenant_activity_audit_logs ENABLE ROW LEVEL SECURITY;`
7. `ALTER TABLE {schema}.tenant_activity_audit_logs FORCE ROW LEVEL SECURITY;`
8. `CREATE POLICY tenant_activity_audit_logs_tenant_isolation ON {schema}.tenant_activity_audit_logs USING (...) WITH CHECK (...);` per the D-29 OR-branch form from pre-flight item 6.
9. Three indexes on tenant table per the design doc Table 1 spec.
10. `CREATE TABLE {schema}.platform_activity_audit_logs ( ... 16 columns ... );` per the design doc Table 2 spec. `tenant_id` and `tenant_name` NULLABLE; all other columns identical to tenant table.
11. FK on platform table's `tenant_id` (same convention; FK accepts NULL values).
12. Two CHECK constraints on platform table (named `ck_platform_activity_audit_logs_resource_pair` and `ck_platform_activity_audit_logs_tenant_pair`): resource_pair as above, plus tenant_pair `(tenant_id IS NULL AND tenant_name IS NULL) OR (tenant_id IS NOT NULL AND tenant_name IS NOT NULL)`.
13. No RLS on platform table (skip ENABLE / FORCE / CREATE POLICY).
14. Two indexes on platform table per the design doc Table 2 spec.

The migration's `downgrade()`:

1. `DROP TABLE {schema}.platform_activity_audit_logs;` (drops indexes and constraints with it).
2. `DROP TABLE {schema}.tenant_activity_audit_logs;` (drops policy, indexes, constraints with it).
3. `DROP TYPE {schema}.audit_result_type_enum;`

Order reversed from upgrade. Reversible round-trip clean.

### Testing and regression discipline

**Load-bearing tests** (failure means the schema is structurally wrong; these MUST pass before commit):

- S1, S2, S3: Migration upgrade / downgrade / round-trip. Without these, the migration is unreliable.
- S6, S7: CHECK constraint enforcement. The CHECKs are the structural invariants of the design (NULL-pair consistency); they must fire.
- S8: FK RESTRICT semantics. The audit-trail-survives-tenant-deletion guarantee depends on this.
- S9: RLS D-29 OR-branch policy correctness. Tenant isolation is load-bearing for the entire subsystem.
- M4: All 6 `AuditResultType` enum values round-trip. The enum vocabulary IS the failure classification surface for sub-steps 6.16.2-5.

**Correctness-only tests** (smaller blast radius; recommended but not blocking):

- S4, S5: Enum value count + reuse verification. Catch misconfiguration; not behaviorally load-bearing once the migration is correct.
