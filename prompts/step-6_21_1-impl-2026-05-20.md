## Context

Frontend reported a bug filed during 5g.1.5: Add Org Node fails when the TENANT row is selected as parent. Investigation (`docs/investigations/2026-05-20-write-surface-coupling.md`, committed earlier in this session) confirmed that the response shape of `GET /api/v1/tenants/{tenant_id}/org-tree` does NOT expose the tenant-root `org_nodes.id`. The frontend, having no other source for that id, synthesised a TENANT row using `data.tenant_id` (the `tenants.id` UUID) and posted it as `parent_id` on Add Org Node. Backend correctly rejected because `tenants.id` and the tenant-root `org_nodes.id` are independent UUIDs.

This step exposes `tenant_root_id`, `tenant_root_code`, and `tenant_root_path` as new top-level fields on `OrgTreeResponse`. Additive, non-breaking. Frontend will use `tenant_root_id` (not `tenant_id`) as the `parent_id` when the synthesized TENANT row is selected. Step 6.21.1 fixes Gap A from the investigation; Gap B is Step 6.21.2.


## Testing and regression discipline

### Load-bearing tests (must catch a real failure mode)

- R1/R2: confirm the new fields land in the response envelope with the correct types and values. If the handler's tenant-root extraction is wrong (wrong predicate, wrong field assignment), R1/R2 fail.
- R3: confirms LD3 doesn't accidentally skip the extraction on empty-tree path. If the handler early-returns when no descendants exist and bypasses the tenant-root scan, R3 fails.
- S1 (cloud smoke): confirms the deployed cloud build serves the new field shape to the frontend. The frontend's Add Org Node fix depends on this assertion passing in cloud.

### Deliberately not added

- **No repo tests.** Per LD2 (handler-side extraction), the repo contract is unchanged. No need to add tests for behaviour that was already verified by existing repo coverage.
- **No test for "tenant 404 still works."** The anchor dependency (`anchor_dep=get_tenant_anchor`) is unchanged and has its own existing test coverage. Re-testing here is duplicate effort.
- **No test for "PLATFORM cross-tenant access."** GET /org-tree's RLS posture is unchanged; existing tests cover this.
- **No test for "PATCH /org-tree still rejects with the new field present in some other endpoint's response."** Out of scope; PATCH /org-tree is unchanged.
- **No EXPLAIN ANALYZE.** No new query is introduced (LD3); query-plan verification not relevant.

### Regression-risk surface

Two failure modes worth specific attention:

1. **The TENANT-typed row not found in the extraction scan.** If the handler's extraction logic uses the wrong predicate (e.g., filters on `parent_id IS NULL` instead of `node_type == TENANT`, or uses a string comparison instead of the enum), the scan returns no match and the handler crashes on the field assignment. R1, R2, R3 all assert the fields land with correct values, which catches this.

2. **OpenAPI description text changes more than expected.** Schema docstring edits sometimes produce broader description diffs than intended (Surface-and-stop #4). Verification harness step 7 explicitly diffs the OpenAPI to surface unexpected description changes.

## Surface-and-stop scenarios

0. **Missing or moved files.** Any file the change list assumes exists (per Pre-flight Checks #2, #3, #5, #9, #10) is missing or at a different path. Stop. Report.

1. **Pre-flight returns unexpected output** other than "missing file" (covered by #0 above). Stop. Report.

2. **Tenant exists but tenant-root org_node is missing at runtime.** Should be impossible post-Step-6.20.1 for production-path tenants, but possible in test fixtures using bare `make_tenant` (default `with_root=False`). The handler's tenant-root extraction returns no match, then crashes on field assignment. Surface: fix R1/R2/R3 fixtures to use `make_tenant(..., with_root=True)`; do NOT relax the new fields to optional. Operator decides if a different fixture path is needed.

3. **OpenAPI description text changes more than expected.** Schema docstring edits sometimes produce broader description diffs than intended. If the regen produces unrelated wording changes (e.g., to other schemas), surface and revert; this step's description scope is `OrgTreeResponse` only (class docstring + `tree` field description).

4. **Empty-tree behaviour.** For a tenant whose only org_node is the TENANT-typed root, the descendants list is empty (after `_build_tree` filters out the TENANT row), so `tree=[]`. The 3 new fields must still populate (extraction runs on the unfiltered row list, before `_build_tree`). If the handler's extraction is placed after `_build_tree` and operates on the filtered tree, the empty-tree case will surface as missing tenant-root data. R3 catches this.

5. **Smoke assertion target tenant differs.** Local smoke and cloud smoke may use different seed tenants. Cloud uses Buc-ee's (stable code `BUC-EES`). Local uses whichever seed tenant the existing `TU_TENANT_ID` resolution finds; use the looser "tenant_root_id is a UUID-shaped string" assertion rather than asserting on the exact code value.

6. **TENANT row appears in `tree[]` instead of being filtered out.** If for any reason the existing `_build_tree` helper changes behaviour and the TENANT row leaks into the response `tree[]`, the schema contract is violated. R1 / R2 / R3 each include an assertion that `tree[]` contains no row with `node_type='TENANT'` to catch this regression.
