## Locked decisions

1. **URL shape: flat `/stores` for list and detail.** Two endpoints; per-tenant scoping via `?tenant_id=` query parameter. Mirrors tenant-users / role-assignments precedent. Owner: design conversation 2026-05-17.

2. **`tenant_name` query posture: LEFT JOIN to `core.tenants`.** Not correlated subquery. Default sort uses the joined column directly. Deviates from tenants resource's correlated-subquery aggregate pattern; rationale is `tenant_name` is a sibling-table label, not an aggregate. Owner: design conversation 2026-05-17.

3. **Sort vocabulary: 8 keys.**
   - `tenant_name_asc` (default; primary `tenants.name ASC`, stable secondary `stores.name ASC`)
   - `tenant_name_desc` (primary `tenants.name DESC`, stable secondary `stores.name ASC`)
   - `name_asc`, `name_desc`
   - `created_at_desc`, `created_at_asc`
   - `status_asc` (DB enum natural order: OPENING, ACTIVE, INACTIVE, CLOSED)
   - `country_asc`

   Naming convention follows the existing codebase pattern (`<field>_asc` / `<field>_desc`); the implicit secondary sort by `stores.name ASC` on the tenant-name keys is documented in the OpenAPI param description, mirroring how tenants documents its stable secondary sort by id. Owner: design conversation 2026-05-17.

4. **Search scope: `name` + `store_code`, case-insensitive substring.** ILIKE `%search%`. Address excluded. Owner: design conversation 2026-05-17.

5. **Filter vocabulary: `tenant_id`, `status`, `country`, `search`.** All optional, all combinable, all AND-ed. No multi-value filters. Owner: design conversation 2026-05-17.

6. **Detail response: 17 fields.** All 22 DDL columns minus 6 audit-actor IDs (`created_by_user_id`, `created_by_user_type`, `updated_by_user_id`, `updated_by_user_type`, `closed_by_user_id`, `closed_by_user_type`), plus `tenant_name`. The `*_at` timestamps stay exposed. Mirrors `TenantRead` precedent. Owner: design conversation 2026-05-17.

7. **Pagination defaults: `offset=0`, `limit=50`, max `limit=100`.** Verify exact numbers against the tenant-users / platform-users routers during implementation; mirror exactly. Owner: design conversation 2026-05-17.

8. **`org_node_id` exposed on detail as a bare UUID column.** No JOIN to `org_nodes.name`. Frontend can fetch the name via `/org-tree` if needed. Owner: design conversation 2026-05-17.

`StoresListRow` and `StoreDetailRow` are repo-internal dataclasses wrapping the joined `(Store, tenant_name)` row pair, mirroring the `TenantListRow` precedent from Step 6.8.3. Router maps to schemas via `_list_item_from_row` / `_detail_from_row` helpers.

List query shape:

```sql
SELECT stores.*, tenants.name AS tenant_name
FROM core.stores
LEFT JOIN core.tenants ON tenants.id = stores.tenant_id
[WHERE ... filters ...]
ORDER BY ... per sort key ...
LIMIT :limit OFFSET :offset
```

LEFT JOIN (not INNER) so a hypothetical orphan store row (no matching tenant) would still surface in the list rather than disappearing. In practice tenant_id is NOT NULL with a foreign key, so the LEFT/INNER distinction never fires; LEFT is defensive.

Total-row-count subquery for the pagination envelope uses the same filter WHERE clause without the JOIN (the JOIN is for the `tenant_name` column only; count is on `stores` alone).

Invalid sort key raises `InvalidSortKeyError` (reuse existing from `repositories/_errors.py`).

Plus two FN-AB additions deferred from 6.17.1:

### FN-AB-NN — Seed loader does not strip whitespace on Excel cells

The Excel-driven seed loader passes cell values through unchanged. A trailing space in an enum-typed cell (e.g., `'VIEW '` instead of `'VIEW'`) causes `psycopg.errors.InvalidTextRepresentation` at INSERT time. Surfaced at Step 6.17.1 when adding new permission rows manually.

**How to apply.** Until the loader is hardened, manually verify enum-typed cells for trailing/leading whitespace before reseed. The loader could add `.strip()` to all cell reads as a defensive fix.

**Resolution criterion.** Either: (a) loader gains `strip()` on all string cell reads (one-line PR), or (b) the convention is to verify whitespace post-edit and the FN-AB stays open as a procedural warning. Either is acceptable; pick at the next loader touch-point.

### FN-AB-NN — `core.lookups` is dual-sourced (migration + SQL delta, not Excel)

The `core.lookups` table is seeded from two sources today: 17 rows from the Step 3.6 Alembic migration (tenant_tier, tenant_region, tenant_status, tenant_industry) and 6 rows from Step 6.17.1's inline SQL delta (store_status, tax_treatment). Excel does NOT have a `lookups` sheet. `seed_dev_data --reset` does not truncate `core.lookups`, so the SQL-managed rows survive reseeds.

**How to apply.** Future lookup additions should follow the same pattern: inline SQL UPSERT on both local and Cloud SQL, idempotent, no Excel sheet. If a Stage 3 or later step demands lookup-table-as-Excel parity, that's a dedicated migration step, not a side effect of another step.

**Resolution criterion.** Documentation only; no code change needed. Closes if/when a dedicated `lookups`-to-Excel migration step ships and Excel becomes authoritative.
