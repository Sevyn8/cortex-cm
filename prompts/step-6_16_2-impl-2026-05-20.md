## Scope in (this commit)

Wire synchronous audit emission into 4 write endpoints on `/api/v1/tenants/*`:
- `POST /api/v1/tenants` (create)
- `PATCH /api/v1/tenants/{tenant_id}` (update)
- `POST /api/v1/tenants/{tenant_id}/suspend`
- `POST /api/v1/tenants/{tenant_id}/activate`

Both success and failure paths emit audit rows per design doc emission contract.

Bundle one design doc refinement: clarify two-rule transaction semantics in `docs/architecture_audit_logs.md` Emission contract section (success = same transaction as data write; failure = separate new transaction after the data transaction has rolled back). This refinement was surfaced during 6.16.2 design Phase 2; landing it in 6.16.2's commit per A6.

### 2. `src/admin_backend/audit/emit.py` (NEW)

The emission module. Contents:

**Module imports.** SQLAlchemy session and engine types; AuthContext from auth module; AuditResultType enum from models.audit_log; ActorUserType enum (existing). UUID + datetime stdlib.

**`AUDITED_ROUTES` constant.** Module-level dict mapping `(method, path_template)` to `(action, resource_type, route_to_platform)`. Initial entries (the 4 tenant endpoints):

```python
AUDITED_ROUTES = {
    ("POST", "/api/v1/tenants"): ("CREATE", "TENANT", True),
    ("PATCH", "/api/v1/tenants/{tenant_id}"): ("UPDATE", "TENANT", False),
    ("POST", "/api/v1/tenants/{tenant_id}/suspend"): ("SUSPEND", "TENANT", False),
    ("POST", "/api/v1/tenants/{tenant_id}/activate"): ("ACTIVATE", "TENANT", False),
}
```

The `route_to_platform` flag captures the design doc's named exception: POST /tenants routes to platform_activity_audit_logs regardless of tenant_id presence.

**`_actor_type_from_auth(auth_user_type)` helper.** Local copy per LD6 (FN-AB-58). Maps Literal['PLATFORM', 'TENANT'] to the SQLAlchemy ActorUserType enum value.

**`_label_for_action(action)` helper.** Maps action codes to display labels: CREATE -> "Created", UPDATE -> "Updated", SUSPEND -> "Suspended", ACTIVATE -> "Activated". Module-level dict.

**`emit_audit_event(session, *, auth, action, action_label, resource_type, resource_id, resource_label, result_type, details, tenant_id, tenant_name, request_id, route_to_platform)` function.** Synchronous, same-transaction emission. Detailed signature:

- `session`: SQLAlchemy Session, the same session the data write is using.
- `auth`: AuthContext.
- `action`, `action_label`, `resource_type`, `resource_id`, `resource_label`, `result_type`, `details`, `tenant_id`, `tenant_name`, `request_id`: per audit row columns.
- `route_to_platform`: bool. If True, route to platform_activity_audit_logs even if tenant_id is set (tenant creation success). If False, route per the routing principle.

Behaviour:
- Construct the appropriate ORM model instance (`TenantActivityAuditLog` or `PlatformActivityAuditLog`).
- Call `session.add(row)` and `session.flush()`. Do NOT commit; the caller controls the transaction.
- On failure, the SQLAlchemy exception bubbles up; the caller's transaction rolls back atomically.

**`emit_audit_event_in_new_transaction(engine, *, ...same kwargs...)` function.** Failure-path emission in a separate transaction. Behaviour:

- Open a new connection via `engine.begin()` (autocommit context).
- INSERT the audit row using raw SQL `text(...)` with schema qualification per CSD-03 pattern (since this path doesn't have access to the ORM session context).
- Commit on context-manager exit.
- If the insert itself fails (rare; constraint violation due to bug), log a CRITICAL message and silently fail. The user-facing error envelope is NOT blocked by audit emission failure on the failure path.

**`build_success_details(...)`, `build_permission_denied_details(...)`, `build_validation_failed_details(...)`, `build_conflict_details(...)`, etc. helper builders.** Each takes the relevant inputs (e.g., snapshot of created entity, required_permission tuple, validation errors list) and produces the JSONB structure per the design doc Failure-row payload shapes section. These are pure functions returning dict[str, Any]; the emission helpers serialise to JSONB at INSERT time.

### 3. `src/admin_backend/errors.py` (or wherever the global exception handler lives; MODIFY per pre-flight Check #4)

Extend the existing global exception handler with audit failure-path emission. Behaviour:

- After determining the error class and constructing the response envelope (existing logic), check if the request's route template + method is in `AUDITED_ROUTES`.
- If yes, extract:
  - Auth context (already in the request scope if auth succeeded; if auth FAILED, auth is None; in that case emission is skipped for v0 since we have no actor : defer to a future step if we want to log unauthenticated attempts).
  - Path params (e.g., tenant_id for PATCH/suspend/activate).
  - Error class (mapped to AuditResultType: ClientError 403 -> PERMISSION_DENIED; 422 -> VALIDATION_FAILED; 409 -> CONFLICT; 500 -> INTERNAL_ERROR; IntegrityError -> INTEGRITY_VIOLATION).
  - Request_id (from request.state.request_id or freshly generated).
- Build the appropriate details payload via the corresponding builder.
- Call `emit_audit_event_in_new_transaction(engine, ...)`. Engine is obtained from FastAPI app state or via a dependency-injection pattern; pre-flight Check #5 surfaces the exact mechanism.
- Continue with the standard error envelope return.

If emission fails (e.g., DB connection issue), log CRITICAL and continue. The original error envelope is the user-visible result; audit emission failure is a back-end-only concern.

### 9. `docs/architecture_audit_logs.md` (MODIFY)

Refine the **Emission contract** section to specify two-rule transaction semantics. The current text says "synchronous, same transaction" as one rule. The refinement:

- Add a paragraph clarifying that the rule applies to SUCCESS rows: data write + audit row commit together in the same transaction.
- Add a paragraph for FAILURE rows: when the data transaction has rolled back, the audit row CANNOT be in that transaction (it was discarded). A separate one-statement transaction writes the audit row after the data transaction has cleared. This is still synchronous (the request waits for the audit emission), still atomic per-row (single statement); just a different transaction.
- Note that this refinement was surfaced during 6.16.2 design Phase 2 and is the canonical interpretation going forward.

## Appendix A: framing prose for `docs/architecture_audit_logs.md` Emission contract refinement (LD15)

Per WORKFLOW.md Architecture doc updates discipline: Phase 2 close drafts the framing prose; Phase 4 folds it into the impl prompt as Appendix A; Phase 5 applies it. The framing prose below is the canonical text to incorporate into `docs/architecture_audit_logs.md` at the **Emission contract > Synchronous, same transaction (v0)** subsection. CC composes the exact insertion location and surrounding edit shape at apply time; the framing content below is the source of truth for what the section now says.

### Refinement framing

The existing section title "Synchronous, same transaction (v0)" presents a single-rule emission contract. The 6.16.2 work surfaces that this contract has two distinct rules depending on path outcome:

1. **Success path: same transaction as the data write.** The data write and the audit row INSERT share one transaction. Both commit together. If either fails, both roll back. This preserves atomicity per the original design intent: a row in the audit log implies the data change happened; absence of an audit row implies no data change happened.

2. **Failure path: separate new transaction after the data transaction has rolled back.** When the data write fails (CHECK constraint, FK violation, app-layer raise), the transaction enters error state and is rolled back. An audit INSERT inside that transaction CANNOT commit; it gets discarded along with the data write. The audit row for the failed attempt must therefore be written in a new, separate transaction opened AFTER the failed one has cleared. This is still synchronous (the request waits for the audit emission), still atomic per-row (a single INSERT statement); just a different transaction.

The two-rule structure is not a deviation from the design intent; it is a clarification of what "same transaction" mechanically means in success vs failure flows. The original "synchronous, same transaction" wording is replaced by section copy that names both rules explicitly.

### Implementation impact (informational; the section itself stays principle-level)

Codebase mechanics implied by the refinement, but NOT included in the architecture doc section (it stays principle-focused):

- Success-path emission: called inside repo methods that own the data write transaction. Same SQLAlchemy session.
- Failure-path emission: triggered from the global exception handler. Opens a new database connection via the engine; runs INSERT; commits the connection's autocommit context.
- If failure-path emission ITSELF fails (rare; constraint violation due to bug), it logs CRITICAL and silently continues. The user-facing error envelope is the visible result; audit emission failure on the failure path is a back-end-only concern (we already lost the data change).

### What this section does NOT say (deliberate scope-out)

- It does not specify which endpoints are audited (that lives in the AUDITED_ROUTES dict, an implementation detail).
- It does not specify async or outbox patterns (deferred per Scale Considerations).
- It does not specify retry policy on failure-path emission failure (CRITICAL log only at v0).
- It does not enumerate every error class to result_type mapping (table covers the result_type taxonomy; mapping is a code concern).

The refined section reads as principle ("here are the two rules"), with mechanics implied but pushed to implementation. CC composes the actual prose at apply time, working from this framing.

- **Routing principle** (the upstream design choice this rule serves).
- **Failure-row payload shapes** (the downstream table that catalogues per-result_type details).
- **Scale considerations > Forward options > Async emission via outbox pattern** (the future-state mitigation when v0 sync emission becomes a latency cost).
