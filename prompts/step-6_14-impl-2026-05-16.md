|---|----------|----------------|
| LD1 | Request body field `roles` shape: `list[RoleAssignmentItem]` where `RoleAssignmentItem = {role_id: UUID, org_node_id: UUID}`. Bare-UUID legacy shape rejected as 422 at Pydantic validation. | LD2 | Tenant root is an ordinary org_node UUID. No null, no sentinel, no server-side default. The frontend resolves tenant-root via the existing `GET /tenants/{id}/org-tree` and sends a real UUID. | request validation; no special handling needed |
| LD3 | Diff-replace logic applied uniformly to both POST and PATCH. Computed against `frozenset[(role_id, org_node_id)]`. Unchanged triples in (current ∩ desired) leave their `tenant_user_role_assignments` row untouched (preserves `granted_at`, `granted_by_*`, `updated_at`). Rows in (current − desired) flip to INACTIVE with `revoked_*` populated. Rows in (desired − current) INSERT as ACTIVE. The 6.10.1 whole-set replace path is retired; this is a one-commit retrofit. | `repositories/tenant_users.py` |
| LD4 | Pre-write validation order (deterministic for testability): (1) Pydantic shape + within-request duplicate check; (2) `roles` referenced exist and are non-ARCHIVED and audience=TENANT; (3) `org_nodes` referenced exist in the target tenant and are non-ARCHIVED; (4) compute current ACTIVE set under SELECT FOR UPDATE; (5) apply diff. Any validation failure rolls back the entire transaction; no partial application. | `repositories/tenant_users.py` |
| LD5 | Within-request duplicate detection: if the same `(role_id, org_node_id)` pair appears more than once in the submitted `roles` list, return 422 `DUPLICATE_ROLE_ASSIGNMENT_IN_REQUEST` from a Pydantic validator. Catches client bugs ahead of the DB UNIQUE-index violation. | `schemas/tenant_user.py` validator |
| LD6 | Archived role or archived org_node validation: 422 `INVALID_ROLE` (covers missing + ARCHIVED uniformly, mirroring 6.10.1's existing behavior for missing roles) or 422 `INVALID_ORG_NODE` (NEW; same shape for missing + ARCHIVED). | LD7 | Concurrent-edit race: if a parallel transaction inserts a duplicate (user, role, org_node) ACTIVE row between our SELECT FOR UPDATE and INSERT, the UNIQUE partial index `uq_tenant_user_role_assignments_active` rejects the second INSERT. ## DDL facts (verified against current_schema.sql at 2026-05-16)

### core.tenant_user_role_assignments

- PK: `id UUID DEFAULT core.uuidv7() NOT NULL`.
- Four FKs:
  - `(tenant_id, tenant_user_id)` to `tenant_users(tenant_id, id)` (D-34 composite, same-tenant enforced).
  - `(tenant_id, org_node_id)` to `org_nodes(tenant_id, id)` (D-34 composite, same-tenant enforced; cross-tenant org_node_id rejected at INSERT).
  - `role_id` to `roles(id)` (single-column; roles is global per D-30 exception).
  - `tenant_id` to `tenants(id)`.
- `org_node_id NOT NULL`. Tenant root is just an org_node like any other.
- **UNIQUE partial index `uq_tenant_user_role_assignments_active`** on `(tenant_user_id, role_id, org_node_id) WHERE status='ACTIVE'`. Licenses Pattern B (same role at distinct anchors); blocks duplicate ACTIVE rows.
- Audit-actor: **Pattern (b)** per D-13. `granted_by_user_id` + `granted_by_user_type`, same for `revoked_by_*`. NO FK from `*_by_user_id` to a users table; both PLATFORM and TENANT actors can grant or revoke.
- Three CHECK constraints:
  - `ck_..._granted_by_actor_pair`: both NULL or both NOT NULL.
  - `ck_..._revoked_by_actor_pair`: both NULL or both NOT NULL.
  - `ck_..._revoked_consistency`: status='INACTIVE' requires all three revoked_* fields populated; status='ACTIVE' requires all three NULL.
- Status enum `core.user_role_assignment_status_enum`: `ACTIVE`, `INACTIVE`.
- Standard `updated_at` (DEFAULT `now()`, NOT NULL).
- RLS: ENABLE + FORCE. Policy `tenant_user_role_assignments_tenant_isolation` USING and WITH CHECK both: `tenant_id = current_setting('app.tenant_id') OR app.user_type='PLATFORM'`. Standard D-29 OR-branch.

### core.roles (referenced, not written)

- Single-column PK; no `tenant_id` column (roles are platform-global; tenants pick from the same catalogue).
- `audience` enum on the role (TENANT or PLATFORM). 6.14's `roles` field accepts TENANT-audience roles only; PLATFORM-audience rejected as 422 `INVALID_ROLE_AUDIENCE` per 6.10.1's existing logic.
- `status` enum includes `ARCHIVED`. CHECK constraint ties `archived_*` fields to status='ARCHIVED'.
- **Implication:** assigning an archived role rejects as 422 `INVALID_ROLE` (aggregated with missing-role cases; same error code per LD6).

### core.org_nodes (referenced, not written)

- Same-tenant composite-FK target.
- `status` enum includes `ARCHIVED`.
- **Implication:** assigning at an archived org_node rejects as 422 `INVALID_ORG_NODE` (aggregated with missing-org_node cases; same error code per LD6).

## Transition contract (per request)

| Current row state | Desired in request | Outcome |
|---|---|---|
| no row | `(role_id, org_node_id)` present | INSERT new ACTIVE row; `granted_at=now()`; `granted_by_*` populated |
| ACTIVE row exists | `(role_id, org_node_id)` present (matches current) | NO WRITE; row preserved verbatim including `granted_at` |
| ACTIVE row exists | `(role_id, org_node_id)` absent (not in desired set) | UPDATE row to INACTIVE; `revoked_at=now()`; `revoked_by_*` populated |
| INACTIVE row exists | `(role_id, org_node_id)` present | INSERT new ACTIVE row (the INACTIVE row stays as audit trail); the UNIQUE partial index permits this since it's WHERE status='ACTIVE' |
| INACTIVE row exists | `(role_id, org_node_id)` absent | NO WRITE; INACTIVE row preserved as-is |
