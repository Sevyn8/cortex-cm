TenantUsersRepo.create(
    session: AsyncSession,
    *,
    tenant_id: UUID,
    email: str,                  # already lowercased by schema validator
    full_name: str,
    role_ids: list[UUID],        # already deduped + audience-checked
    actor_user_id: UUID,
    actor_user_type: ActorUserType,  # PLATFORM | TENANT
) -> TenantUserDetailRow

  Behavior (single transaction):
    1. Lookup tenant root org_node:
         SELECT id FROM core.org_nodes
          WHERE tenant_id = :t AND node_type = 'TENANT'
       - 0 rows → raise TenantNotFoundError (RLS-as-404)
       - Should always be exactly 1 row given ck_root_parent_consistency
    2. Pre-check email uniqueness within tenant:
         SELECT 1 FROM core.tenant_users
          WHERE tenant_id = :t AND email = :e LIMIT 1
       - hit → DuplicateTenantUserEmailError
    3. INSERT core.tenant_users:
         - status = 'INVITED'
         - auth0_sub = NULL
         - invited_at = NOW()
         - invitation_accepted_at = NULL
         - created_by_user_id, created_by_user_type populated
         - updated_by_user_id, updated_by_user_type populated
         - RETURNING id
    4. For each role_id in role_ids:
         INSERT core.tenant_user_role_assignments:
           - tenant_user_id = <new_user_id>
           - tenant_id = :t            (composite-FK enforces same-tenant)
           - org_node_id = <tenant_root_id>
           - role_id = :role_id
           - status = 'ACTIVE'
           - granted_at = NOW()
           - granted_by_user_id, granted_by_user_type populated
         The audience-check trigger fires per row; schema pre-check
         should have already caught audience mismatches.
    5. SELECT-with-aggregates to build TenantUserDetailRow.
       Reuse the existing pattern in `tenant_users.py` for inline
       roles[] (6.8.3's jsonb_agg subquery).

TenantUsersRepo.update(
    session: AsyncSession,
    user_id: UUID,
    *,
    fields: dict[str, Any],      # only PATCH-allowed fields per Pydantic
    actor_user_id: UUID,
    actor_user_type: ActorUserType,
) -> TenantUserDetailRow | None

  Behavior:
    None → 404 (RLS-as-404 per D-17). Caller handler raises
    TenantUserNotFoundError on None.

    If 'email' in fields:
      - Pre-check uniqueness within tenant excluding self:
          SELECT 1 FROM core.tenant_users
           WHERE tenant_id = :t AND email = :e AND id != :uid LIMIT 1
      - hit → DuplicateTenantUserEmailError
      - Email is already lowercased by Pydantic validator.

    If 'roles' in fields (replace-set semantics):
      - role_ids already audience-checked by Pydantic.
      - Bulk UPDATE existing ACTIVE assignments to INACTIVE:
          UPDATE core.tenant_user_role_assignments
             SET status = 'INACTIVE',
                 revoked_at = NOW(),
                 revoked_by_user_id = :actor,
                 revoked_by_user_type = :actor_type
           WHERE tenant_user_id = :uid AND status = 'ACTIVE';
      - INSERT new ACTIVE assignments for the new role set
        (same shape as create's step 4).
      - If new role set equals existing ACTIVE set, the UPDATE
        revokes-then-INSERTs unnecessarily. Optimization (skip
        if equal) is acceptable but not required — the
        partial-unique uq_active will not collide because the
        revoke step flips status before the INSERT step.

    Always bump updated_at + updated_by_user_id + updated_by_user_type.

    Return the post-update TenantUserDetailRow.

TenantUsersRepo.transition(
    session: AsyncSession,
    user_id: UUID,
    *,
    target_status: Literal["SUSPENDED", "ACTIVE"],
    actor_user_id: UUID,
    actor_user_type: ActorUserType,
) -> tuple[TenantUserDetailRow | None, TransitionResult]

  Behavior:
    1. Read current status FOR UPDATE:
         SELECT status FROM core.tenant_users
          WHERE id = :uid FOR UPDATE
       - 0 rows → (None, TransitionResult.NOT_FOUND)
    2. Validate transition matrix:
         current=ACTIVE, target=SUSPENDED → OK; set suspended_*
         current=SUSPENDED, target=ACTIVE → OK; clear suspended_*
         Any other source → INVALID_STATE_TRANSITION
       INVITED → SUSPENDED is structurally rejected by
       ck_auth0_sub_consistency; mapping it to
       INVALID_STATE_TRANSITION at the app layer means the user
       never sees a 500.
    3. UPDATE with status flip + suspended_* set/clear +
       updated_by_*  + suspended_by_*  pair population:
         For ACTIVE→SUSPENDED:
           SET status = 'SUSPENDED',
               suspended_at = NOW(),
               suspended_by_user_id = :actor,
               suspended_by_user_type = :actor_type,
               updated_by_user_id = :actor,
               updated_by_user_type = :actor_type
         For SUSPENDED→ACTIVE:
           SET status = 'ACTIVE',
               suspended_at = NULL,
               suspended_by_user_id = NULL,
               suspended_by_user_type = NULL,
               updated_by_user_id = :actor,
               updated_by_user_type = :actor_type
       (set_updated_at trigger handles updated_at.)
    4. SELECT-with-aggregates for return shape.

  Returns: (row, TransitionResult.OK) on success.
```

**Self-edit guard placement.** The guard runs **after** Layer 2
`has_permission` passes (the user has CONFIGURE.TENANT) but **before**
the repository call. It is handler-layer logic, not gate logic — the
gate factory has no knowledge of "subject of action vs actor of
action." The guard fires for TENANT callers only; a PLATFORM
SUPER_ADMIN editing a tenant user cannot be self-editing
(PLATFORM users live in `platform_users`, a different table).

### Worked example: POST /tenant-users/{user_id}/suspend and /activate (multi-audience transitions)

\`\`\`python
@router.post(
    "/{user_id}/suspend",
    response_model=TenantUserDetailResponse,
)
async def suspend_tenant_user(
    user_id: UUID,
    _: None = Depends(require(
        ModuleCode.ADMIN,
        PermissionResource.USERS,
        PermissionAction.CONFIGURE,
        PermissionScope.TENANT,
        anchor_dep=get_tenant_user_anchor,
    )),
    auth: AuthContext = Depends(get_auth_context),
    session: AsyncSession = Depends(get_tenant_session_dep),
) -> TenantUserDetailResponse:
    \"\"\"Suspend a tenant user. Multi-audience.

    Allowed source: ACTIVE only.
      - INVITED → SUSPENDED is structurally impossible
        (ck_auth0_sub_consistency rejects).
      - SUSPENDED → SUSPENDED returns 409 INVALID_STATE_TRANSITION.

    Self-suspend forbidden (TENANT-audience self-edit guard).
    \"\"\"
    if auth.user_type == "TENANT" and auth.user_id == user_id:
        raise SelfEditForbiddenError()

    user_row, result = await TenantUsersRepo.transition(
        session=session,
        user_id=user_id,
        target_status="SUSPENDED",
        actor_user_id=auth.user_id,
        actor_user_type=auth.user_type,
    )
    if result == TransitionResult.NOT_FOUND:
        raise TenantUserNotFoundError()
    if result == TransitionResult.INVALID_STATE_TRANSITION:
        raise InvalidStateTransitionError()
    return _to_response(user_row)
\`\`\`

**Transition lifecycle** for `core.tenant_users`:

\`\`\`
[create] ──► INVITED ──[Auth0 accept; out of scope]──► ACTIVE
                                                       ▲ │
                                                  activate │ suspend
                                                       │ ▼
                                                    SUSPENDED
\`\`\`

- `INVITED → SUSPENDED` rejected by `ck_auth0_sub_consistency` (SUSPENDED
  requires `auth0_sub IS NOT NULL`; INVITED requires `auth0_sub IS NULL`).
  App layer maps the rejection to 409 INVALID_STATE_TRANSITION.
- `SUSPENDED → ACTIVE` requires clearing `suspended_at`,
  `suspended_by_user_id`, and `suspended_by_user_type` to NULL,
  atomically with the status flip, to satisfy
  `ck_suspended_consistency`.
- `ACTIVE → SUSPENDED` requires setting all three suspended_*
  columns to non-NULL.
```

```markdown
### Pattern (b) audit-actor population — repo-side requirement

Tables using Pattern (b) audit-actor columns (per D-13: tenant_users,
org_nodes, stores, roles, role_permissions, both role_assignments
tables) require that repository INSERT/UPDATE statements populate
BOTH columns of each `*_by_user_*` pair simultaneously. Each table
carries a `ck_*_actor_pair` CHECK constraint enforcing either
both-NULL or both-NOT-NULL.

Example: writing to `core.tenant_users` requires:

\`\`\`sql
INSERT INTO core.tenant_users (
    ...,
    created_by_user_id,     -- UUID
    created_by_user_type,   -- actor_user_type_enum
    updated_by_user_id,     -- UUID
    updated_by_user_type    -- actor_user_type_enum
) VALUES (
    ...,
    :actor_user_id,
    :actor_user_type,
    :actor_user_id,
    :actor_user_type
);
\`\`\`

Common mistake: writing only `*_by_user_id` and leaving
`*_by_user_type` NULL violates the pair-CHECK. The repo's bound
parameter dict must include the actor_user_type alongside
actor_user_id on every Pattern (b) write.

This is distinct from Pattern (a) tables (tenants, platform_users)
where the audit-actor column is a single typed FK to `platform_users`
and no `*_user_type` column exists.
```
