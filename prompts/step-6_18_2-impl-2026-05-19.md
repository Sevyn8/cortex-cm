## Locked decisions

**LD1: Response shape is `RoleDetail` with both held and available permissions embedded.**

```
RoleDetail:
  id, name, code, description, audience, status, is_system, user_count,
  created_at, updated_at,
  permissions: list[PermissionDetail]            (held by this role)
  available_permissions: list[PermissionDetail]  (grantable to role; audience-filtered)
```

Source: Phase 2 design Q-new-4 (operator chose option c with later refinement to embed both held and available so frontend renders edit screen from one URL).

**LD2: Available_permissions audience filter applied at repo SQL.**

For PLATFORM-audience roles: `available_permissions` = catalogue minus held. No scope filter.

For TENANT-audience roles: `available_permissions` = catalogue minus held, with additional `WHERE scope != 'GLOBAL'`. Per LD17 (audience-scope coherence rule).

Source: Phase 2 LD17.

**LD3: `PermissionDetail` is a new schema, not an extension of `PermissionRead`.**

`PermissionRead` (used by GET /permissions and GET /roles/{id}/permissions) stays unchanged. `PermissionDetail` adds 4 label fields (module_label, resource_label, action_label, scope_label).

Source: investigation report Bucket 4 + operator decision (Phase 2 closure).

**LD4: Label resolution via 4 LEFT JOINs on `core.lookups`.**

Mirror the existing pattern from `repositories/permission_matrix.py:128-158`. COALESCE fallback to the enum code string if a lookup row is missing.

Lookup list_names (per investigation Bucket 1c):
- `module_code` (for module_label)
- `resource` (for resource_label)
- `permission_action` (for action_label)
- `permission_scope` (for scope_label)

Source: investigation Bucket 1b.

**LD5: Audience filter at repo for 404-on-cross-probe.**

TENANT JWT probing a PLATFORM role's id receives 404 ROLE_NOT_FOUND. Mirrors the existing `RolesRepo.get_by_id` pattern (cited Step 6.1 precedent, repositories/roles.py). The handler calls the new `get_detail_by_id` with the same `audience_filter` shape.

Helper: `_audience_filter_for(auth)` already exists in `routers/v1/rbac.py` (Step 6.1 reuse).

Source: D-17 / D-29 RLS-as-404 anti-information-disclosure pattern.

**LD6: GATE_EXEMPT, no permission gate.**

Per Q-new-3 (operator decision: defer FN-AB-30 read-gate hardening). New endpoint joins the existing 4 exempt role endpoints.

Source: operator decision Phase 1 Q-new-3.

**LD7: Pagination not needed; no limit/offset on available_permissions.**

Catalogue is bounded (36 permissions today; ~50 at scale). Single response carries everything. Mirrors how `RolePermissionsResponse` returns all role permissions without pagination.

Source: Step 6.1 precedent (`RolePermissionsResponse`).

**LD8: Sort order for permissions and available_permissions is by `code` ascending.**

Stable, alphabetical, frontend can re-group by module/resource as needed. Mirrors `list_permissions_for_role`'s existing ORDER BY (module/resource/action/scope ascending, which is equivalent to code-ascending when codes follow the dot-tuple format).

Source: Step 6.1 precedent (cited from `repositories/roles.py::list_permissions_for_role` ORDER BY).

**LD9: `user_count` reused from existing `_user_count_subquery` pattern.**

The new `get_detail_by_id` uses the same correlated subquery pattern as `list_grouped`. RLS on `user_role_assignments` automatically scopes count to caller's audience.

Source: Step 6.1 precedent (cited from `repositories/roles.py`; verify exact pattern at pre-flight).

### 4. `src/admin_backend/repositories/roles.py` (MODIFY: +get_detail_by_id)

Add a new method. Approach: 3 queries inside a single async session call:

1. `role_row` = SELECT role + user_count (correlated subquery) + audience filter. Returns None if role doesn't exist or audience excludes.
2. `held_permissions` = SELECT permissions JOIN role_permissions WHERE role_id = :role_id, with 4 LEFT JOINs on lookups for labels. Mirror permission_matrix.py SQL pattern.
3. `available_permissions` = SELECT permissions WHERE id NOT IN (held set) [AND scope != 'GLOBAL' if role.audience='TENANT'], with same 4 LEFT JOINs.

Pseudocode:

```python
async def get_detail_by_id(
    self,
    session: AsyncSession,
    role_id: UUID,
    *,
    audience_filter: str | None,
) -> dict | None:
    """Return role + user_count + held permissions (with labels) +
    available permissions (with labels, audience-filtered).

    Returns None if role doesn't exist OR audience_filter excludes it
    (RLS-as-404 anti-information-disclosure).

    Returns dict shaped for direct RoleDetail.model_validate(...) by
    the handler.
    """
    # Step 1: fetch role + user_count, applying audience filter.
    # Mirror get_by_id pattern. If None, return None.

    # Step 2: fetch held permissions with labels.
    # Raw SQL with 4 LEFT JOINs on lookups (mirror permission_matrix.py:128-158).

    # Step 3: fetch available permissions with labels.
    # Same 4 LEFT JOIN pattern; WHERE id NOT IN (held).
    # If role.audience == 'TENANT': also WHERE scope != 'GLOBAL'.

    return {
        "id": role.id,
        "name": role.name,
        "code": role.code,
        "description": role.description,
        "audience": role.audience,
        "status": role.status,
        "is_system": role.is_system,
        "user_count": user_count,
        "created_at": role.created_at,
        "updated_at": role.updated_at,
        "permissions": held_permissions,        # list of PermissionDetail-shaped dicts
        "available_permissions": available_permissions,
    }
```

Schema-qualify raw SQL: `core.roles`, `core.permissions`, `core.role_permissions`, `core.lookups`. Per CSD-03.

**Consider: extract the 4-LEFT-JOIN label query as a shared helper.** Either inside `repositories/roles.py` (local function `_select_permissions_with_labels(role_id, audience_filter, only_held)`) or as a method on `PermissionMatrixRepo` (since it already has the pattern). Surface the choice at pre-flight; mirror whichever fits Step 6.1 precedent best. Prefer DRY (one place for the JOIN SQL).

### 5. `src/admin_backend/routers/v1/rbac.py` (MODIFY: +get_role handler)

Add a new handler under the existing `roles_router`:

```python
@roles_router.get(
    "/{role_id}",
    response_model=RoleDetail,
    summary="Get role detail with held + available permissions",
)
async def get_role(
    role_id: UUID,
    auth: AuthContext = Depends(get_auth_context),
    session: AsyncSession = Depends(get_tenant_session_dep),
) -> RoleDetail:
    """E7: self-contained role detail for the edit screen.

    Returns RoleDetail carrying role metadata + held permissions
    (with display labels) + grantable permissions (catalogue minus
    held; TENANT roles exclude GLOBAL-scope per LD17 audience-scope
    coherence rule).

    Audience filter: TENANT JWT probing a PLATFORM role's id receives
    404 ROLE_NOT_FOUND (RLS-as-404 anti-information-disclosure).

    repo = RolesRepo()
    detail = await repo.get_detail_by_id(
        session, role_id, audience_filter=_audience_filter_for(auth),
    )
    if detail is None:
        raise RoleNotFoundError()
    return RoleDetail.model_validate(detail)
```
