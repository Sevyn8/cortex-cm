### 1. `src/admin_backend/audit/emit.py` (MODIFY)

Three changes:

- **`_label_for_action` extension**: ensure SUSPEND / ACTIVATE / UPDATE / CREATE all map to their labels. If 6.16.2 already covered all four, no change needed. Pre-flight Check #4 confirms.

- **`actor_display_name` resolution extension** (conditional on pre-flight Check #6): if the existing helper hardcodes `platform_users`, extend to dispatch on `actor_user_type`. Read from `tenant_users.full_name` when `actor_user_type == TENANT`. Schema-qualify per CSD-03 (`core.tenant_users` / `core.platform_users`).

- **Details-builder extensions** (per LD8 / LD10 / LD11 / LD12). The existing 6.16.2 builders (`build_success_details` and the per-failure-type builders) cover the standard shapes. 6.16.4 adds the new sub-key cases. Match the existing builder-naming convention (verify at pre-flight Check #4). Behaviors required:

  - The SUCCESS builder for **CREATE-shaped** writes accepts a `snapshot: dict` plus optional `roles: list[dict]` and produces `{snapshot: {**snapshot, roles: roles}}` when roles provided. Each role item carries `{role_id, role_name, org_node_id, org_node_name}` per LD9 frozen-label contract.
  - The SUCCESS builder for **UPDATE-shaped** writes accepts `before: dict` and `after: dict` and produces `{before, after}`. Optional `roles` key present on both sides as full list when role assignments changed in this request (Phase 1 Q1 contract). For roles PATCH, the equivalent key is `permissions` on both sides, item shape `{permission_id, permission_code}`.
  - The SUCCESS builder for **transitions** (SUSPEND / ACTIVATE) accepts `before_status: str` and `after_status: str` and produces `{before: {status: before_status}, after: {status: after_status}}`. This matches the 6.16.2 transition payload shape exactly; verify whether 6.16.2 already implements this via the generic UPDATE builder or as a separate function, and follow that precedent.
  - The PERMISSION_DENIED builder accepts an optional `denial_reason: str` kwarg. When present, includes it in the output alongside the standard `{required_permission, caller_audience, caller_roles}` keys. Default omitted (standard PERMISSION_DENIED shape unchanged for 6.16.2 callers).
  - The INTERNAL_ERROR builder accepts an optional `invariant: str` kwarg. When present, includes it alongside the standard `{error_class, sanitised_message}` keys.

  If the existing 6.16.2 builders' signatures cannot cleanly accept these new optional kwargs without breaking call sites, surface and propose either a wrapper builder or a signature evolution.

Builders return JSONB-serializable dicts. Existing 6.16.2 builders reused as-is for the standard shapes.

### Edit 2: optional-sub-key convention in Failure-row payload shapes

**Where**: `docs/architecture_audit_logs.md` Emission contract section, Failure-row payload shapes sub-section.

**Framing**: the 6.16.2 work established standard `details` payload shapes per `result_type` (PERMISSION_DENIED carries `{required_permission, caller_audience, caller_roles}`; INTERNAL_ERROR carries `{error_class, sanitised_message}`; etc.). The 6.16.4 work introduces two cases where the standard shape needs additional context to read clearly in the auditor's UI:

1. **PERMISSION_DENIED from a handler-side guard**: a 403 raised by `_raise_if_self_edit` (Step 6.10.1) is mechanically a permission denial but stems from a different cause than the standard "caller lacked the gate". Adding a `denial_reason` sub-key (value: `"SELF_EDIT_FORBIDDEN"` for this guard) lets the auditor read the specific reason without inferring it from actor_user_id-resource_id equality.

2. **INTERNAL_ERROR from a Layer 2 tripwire**: roles PATCH carries a post-write invariant guard (Step 6.18.3 LD6) that raises `InternalInvariantViolationError` when Layer 1's pre-check has a bug. Adding an `invariant` sub-key (value: the specific guard name, e.g., `"OVERRIDE_GLOBAL_HOLDER_PRESERVATION"`) lets the on-call engineer know which guard fired without grepping the error class name.

The pattern generalises: standard `result_type` shapes admit optional sub-keys naming a specific guard or reason when multiple categories share one `result_type`. The standard sub-keys (the ones in the result_type table) remain present; the optional ones augment.

Future work (6.16.5, post-v0 emission expansion) is free to introduce additional optional sub-keys following this convention. The design doc edit adds one paragraph documenting the convention with the two worked examples; the result_type table itself stays as-is.

CC composes the exact prose at apply time.
