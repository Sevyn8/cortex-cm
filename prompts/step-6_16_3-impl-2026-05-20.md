### 3. `src/admin_backend/repositories/audit_logs.py` (NEW)

New `AuditLogsRepo` class. Two methods:

**`list(session, *, cursor, limit, from_ts, to_ts, status, tenant_id, scope, search, user_type) -> ListResult`**

Where `ListResult` is a small dataclass or NamedTuple: `(items: list[AuditActivityDetailRow], next_cursor: str | None, prev_cursor: str | None, has_more: bool)`. Each `AuditActivityDetailRow` carries all 16 columns plus the synthesised `scope` value; the router-side mapper projects to the 8-field `AuditActivityListItem` for the wire response.

Behaviour:
- Decode cursor if provided (LD13). Raise `InvalidCursorError` on malformed.
- Dispatch on `user_type`:
  - `TENANT`: SELECT from `core.tenant_activity_audit_logs` only. RLS does the tenant_id scoping; no app-layer filter needed. `tenant_id` and `scope` query-param filters silently ignored (LD14, LD15).
  - `PLATFORM`: UNION ALL across both tables with synthesised `scope` column ('PLATFORM' for platform branch, 'TENANT' for tenant branch). Apply filters: `tenant_id` narrows to one tenant on the tenant branch (no rows on platform branch if filter set); `scope` narrows to one branch.
- Apply common filters: `from_ts`, `to_ts`, `status`. Search runs `ILIKE %term%` on 4 columns with OR composition (LD6).
- ORDER BY timestamp DESC, id DESC.
- LIMIT `limit + 1` (one extra to detect `has_more`).
- If `limit + 1` rows returned, drop the last and set `has_more = True`.
- Construct `next_cursor` from the LAST row of the page; `prev_cursor` from the FIRST row of the page (or None on first page).

**`get_by_id(session, *, audit_row_id) -> AuditActivityDetailRow | None`**

Behaviour:
- SELECT from `core.tenant_activity_audit_logs WHERE id = :id LIMIT 1`. If found, return.
- Else SELECT from `core.platform_activity_audit_logs WHERE id = :id LIMIT 1`. If found, return.
- Else return None (router converts to 404 via `AuditEventNotFoundError`).

RLS does the right thing on both queries:
- Tenant caller probing a row in their own tenant: tenant table SELECT succeeds.
- Tenant caller probing a row in another tenant: tenant table SELECT returns 0 rows (RLS filter); platform table is unreachable from tenant API layer. Both branches fail → 404.
- Platform caller: both branches reachable; first match wins.

**Helpers (module-level, prefixed `_`):**

- `_encode_cursor(timestamp, id) -> str`: produces opaque base64.
- `_decode_cursor(cursor: str) -> tuple[datetime, UUID]`: decodes; raises `InvalidCursorError` on any failure.
- `_build_tenant_branch_sql(filters, search) -> str + params`: composes the SELECT for the tenant table branch.
- `_build_platform_branch_sql(filters, search) -> str + params`: composes the SELECT for the platform table branch.
- `_apply_common_filters(sql, params, ...)`: shared filter logic across both branches.
- `_search_clause_sql(...)`: shared search clause builder.

All SQL is raw `text()` with schema-qualified table names per CSD-03 (`f"{schema}.tenant_activity_audit_logs"` etc.). Schema captured per request via existing session GUC; the function uses `get_settings().db_schema` interpolation per the standing convention.

### 7. `tests/integration/test_audit_router.py` (NEW; ~18 router tests)

Naming convention `L1`-`Ln` for list tests, `D1`-`Dn` for detail tests, `P1`-`Pn` for permission/gate tests.

**List tests (L-series):**

- L1: PLATFORM JWT, no filters → 200 with merged stream (both tables), newest-first, default limit 50. LOAD-BEARING.
- L2: TENANT JWT, no filters → 200 with own-tenant rows only (RLS-enforced); scope=TENANT for every row. LOAD-BEARING.
- L3: PLATFORM JWT, `?limit=10` → 200 with 10 items + cursor for next page. LOAD-BEARING.
- L4: Cursor round-trip: page 1 returns next_cursor; page 2 (passing that cursor) returns the next set; no row appears twice; chronological order preserved. LOAD-BEARING.
- L5: PLATFORM JWT, `?status=PERMISSION_DENIED` → only PERMISSION_DENIED rows.
- L6: PLATFORM JWT, `?tenant_id=<id>` → only that tenant's rows + zero platform-branch rows.
- L7: TENANT JWT, `?tenant_id=<other_tenant>` → other tenant's id silently ignored; caller still sees only own-tenant rows (RLS).

### Framing prose

The existing Read contract > Pagination section already specifies cursor pagination as the chosen pattern with `(timestamp DESC, id DESC)` sort key, sequential next/previous navigation, and the three reasons (performance constant at scale; audit timelines feed-shaped; resilience to new events arriving mid-paging). The 6.16.3 work adds one short paragraph noting the deviation from the project's standard offset-based `Pagination` schema, explaining the rationale.

### Suggested insertion point

End of the existing Pagination subsection, before the next subsection (Frontend display). A short paragraph (5-7 lines) along these lines:

```
The audit log subsystem uses cursor pagination, deviating from the project's standard offset-based Pagination schema used elsewhere (tenants, stores, tenant-users). The deviation is intentional: audit log is the only table in the system with structurally unbounded growth, and offset pagination degrades visibly past ~100k rows (the database reads offset + limit rows to return limit rows). Other resources (tenants, stores, tenant-users) are bounded by business scope and offset pagination remains the right pattern there. The CursorPagination response envelope is local to the audit endpoints at v0; if a second cursor-paginated endpoint ships in the future, the envelope promotes to docs/schema or schemas/_common.
```

CC composes the actual insertion at apply time, matching the surrounding prose voice. The framing above is the load-bearing content.
