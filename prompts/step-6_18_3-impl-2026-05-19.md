## Locked decisions

**LD1: PATCH editable scope = name + description + permission_ids only.**

`extra="forbid"` rejects audience, code, is_system, status, audit columns. Empty body returns 422 EMPTY_PATCH (reuse from 6.17.3).

Source: Phase 1 Q-new-1.

**LD2: Audience is immutable on PATCH.**

Trigger-enforced consistency (tg_platform_user_role_assignments_audience_check, tg_tenant_user_role_assignments_audience_check) would break if audience flipped on a role with existing assignments. Schema rejects via extra="forbid".

Source: Phase 1 Q-new-1 follow-up A; investigation Bucket 6a triggers.

**LD3: ARCHIVED roles reject PATCH with 409 ROLE_ARCHIVED.**

Status transitions go through a separate (not-yet-shipped) activate/deactivate API. PATCH is for content edits only.

Source: Phase 1 Q-new-2.

**LD4: Gate is `Depends(require(ADMIN, ROLES, OVERRIDE, GLOBAL))`.**

PLATFORM-only by construction: no TENANT role can hold a `*.GLOBAL` permission per LD17 audience-scope coherence rule + investigation Bucket 2d (confirmed: no TENANT role today holds any `.GLOBAL` permission).

No `audience=` kwarg on the gate. Verify codebase precedent at pre-flight (Check #5).

Source: Phase 1 Q-clarify-1; Phase 2 LD16.

**LD5: Diff-replace on `role_permissions`, not whole-set wipe-and-insert.**

Preserves `created_at` and `created_by_*` audit trail on unchanged rows. Mirrors Step 6.14 precedent (FN-AB-45).

Repo logic:
1. SELECT current `(role_id, permission_id)` rows for the target role.
2. Compute set diff against new `permission_ids` list.
3. DELETE rows in `current - new`.
4. INSERT rows in `new - current` with Pattern (b) audit-actor pair from AuthContext (created_by_user_id + created_by_user_type).
5. Rows in `current ∩ new` untouched (preserve created_at + created_by_*).

Source: Phase 1 Q-new-5; investigation Bucket 3c (Pattern (b) audit-actor columns confirmed); 6.14 retro.

**LD6: Two-layer OVERRIDE.GLOBAL invariant protection.**

**Layer 1 (pre-check, app-facing):**
- Runs ONLY if the proposed edit removes OVERRIDE.GLOBAL from the role-under-edit (LD7 optimization).
- Counts ACTIVE holders of ADMIN.ROLES.OVERRIDE.GLOBAL across all other roles (excluding role-under-edit).
- Returns 409 LAST_OVERRIDE_HOLDER if count would be zero post-edit.
- No write attempted.

**Layer 2 (post-write tripwire, internal):**
- Runs after diff applies, before commit.
- Same invariant query; should always pass if Layer 1 logic is correct.
- If fails: ROLLBACK, raise InternalInvariantViolationError. Class name is for log clarity only; wire returns 500 INTERNAL_ERROR (ServerError convention per investigation Bucket 2c).

Source: Phase 1 (Pattern 1 selection + two-layer refinement); investigation Bucket 6c (corrected invariant SQL).

**LD7: Invariant query filters BOTH assignment-side `status='ACTIVE'` AND user-side `status='ACTIVE'`.**

Filters:
- `pura.status='ACTIVE' AND pu.status='ACTIVE'` (platform side)
- `tura.status='ACTIVE' AND tu.status='ACTIVE'` (tenant side)

Critical correction from investigation Bucket 6c. Without assignment-side filter, INACTIVE (revoked) assignments would count as "active holders" and a true last-holder edit could pass the gate falsely.

Source: investigation Bucket 6c.

**LD8: Layer 1 invariant query excludes the role-under-edit.**

`WHERE rp.role_id != :editing_role_id` in the holder-count CTE. Computes post-edit state by asking "would any OTHER role still have an active holder of OVERRIDE.GLOBAL?"

Layer 2 query does NOT exclude; it reads actual committed-to-be state directly.

Source: Phase 1 LD11; race-condition discussion.

**LD9: Pre-check optimization: invariant query runs only if edit removes OVERRIDE.GLOBAL from current holder.**

If the edit doesn't touch OVERRIDE.GLOBAL on the role-under-edit, the invariant is logically unchanged or strengthened; skip the Layer 1 query for the happy path.

Logic in repo:
```
override_id = (SELECT id FROM permissions WHERE code = OVERRIDE_GLOBAL_CODE)
role_currently_holds_override = override_id in current_perm_ids
role_will_hold_override = override_id in new_perm_ids
if role_currently_holds_override and not role_will_hold_override:
    # Run Layer 1 invariant query.
```

Source: Phase 1 LD10.

**LD10: Audience-scope coherence check (LD17 from Phase 2) at 422 AUDIENCE_SCOPE_MISMATCH.**

For TENANT-audience role under edit: if the new permission_ids set adds any permission with `scope='GLOBAL'` (the diff `new - current`), reject with 422 AUDIENCE_SCOPE_MISMATCH.

Lenient validation: only checks added permissions, not the full current set. Runs BEFORE the OVERRIDE invariant Layer 1 (cheaper, structural-vs-dynamic).

Status code 422 matches `InvalidRoleAudienceError` convention from Step 6.10.1.

Source: Phase 2 LD17 + investigation Bucket 2c.

**LD11: Permission existence pre-check at 422 INVALID_PERMISSION_ID.**

Before audience-scope check: SELECT permissions WHERE id = ANY(:permission_ids). If count mismatch, raise InvalidPermissionError with `missing_ids` in `exc.context`. Body details: null per Q7 envelope lock.

Mirrors Step 6.14 `_validate_roles` + `_validate_org_nodes` pattern. Status 422 matches `InvalidRoleError`, `InvalidOrgNodeError` convention.

Source: Phase 2 LD19 + investigation Bucket 2c naming alignment.

**LD12: SUPER_ADMIN protection at repo (v0 lockout).**

After role fetch + audience/status checks: if `role.code == 'SUPER_ADMIN'`, raise SuperAdminProtectedError (409 SUPER_ADMIN_PROTECTED). No name change, no description change, no permission change via API. Operator must use direct SQL on `core.roles` / `core.role_permissions` for any SUPER_ADMIN edit.

v1 promotion deferred; FN-AB tracks. Code comment at the check site explains the v0 decision.

**LD17: Pre-check + diff order in repo `update` method.**

Order (each step can short-circuit with its named error):

1. SELECT role (404 ROLE_NOT_FOUND if not found, audience filter applied)
2. Check `role.code == 'SUPER_ADMIN'` → SuperAdminProtectedError
3. Check `role.status == 'ARCHIVED'` → RoleArchivedError
4. `body.model_dump(exclude_unset=True)` → EmptyPatchError if empty
5. If `permission_ids` in body:
   a. Fetch permissions by id; missing_ids → InvalidPermissionError
   b. SELECT current role_permissions; compute diff (added, removed, unchanged)
   c. Audience-scope check (LD10) → AudienceScopeMismatchError
   d. Layer 1 OVERRIDE invariant (LD6, LD8, LD9) → LastOverrideHolderError
6. Apply UPDATE on roles row (name/description/updated_by_*)
7. Apply DELETE on removed role_permissions rows
8. Apply INSERT on added role_permissions rows (with audit-actor LD14)
9. Layer 2 OVERRIDE invariant tripwire (LD6) → InternalInvariantViolationError + ROLLBACK
10. Commit
11. Re-fetch RoleDetail via `get_detail_by_id` for response

Source: Phase 2 design block; reordering for early-exit and security-criticality.
